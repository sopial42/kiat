## 1. Philosophy and position in the pyramid

> **Core principle: every vertical-slice story ships a real-backend E2E spec for its happy path. No exceptions.** A vertical slice claims to deliver a user-observable increment end-to-end. If there's no real-backend E2E test proving that increment works against the full stack (browser → frontend → backend → real DB, external APIs mocked via Smocker), the claim is unverified. This rule is enforced at three levels: the spec writer describes the scenario (`## Test scenarios → ### Real-backend E2E`), the coder implements it under `e2e/real-backend/`, and the reviewer blocks if it's missing.

Playwright is the **outer ring** of the pyramid. It validates that:

- The frontend renders correctly for real browser users
- Auth actually works end-to-end (middleware + client hooks + backend JWT)
- The frontend-backend wiring carries the right headers, polls at the right cadence, and reacts to the right responses
- Business flows (signup → action → result) complete without silent breakage

It is **NOT** a replacement for:
- Unit tests (Go service layer, React hooks logic)
- Venom API tests (HTTP contract against a real backend)

**Expected budget:** ~10% of total test count, ~40% of CI wall-clock time. If either number grows, push complexity down to lower rings.

**Two test categories, two directories:**

| Directory | Target backend | When to use |
|---|---|---|
| `frontend/e2e/` | Mocked via `page.route()` | Error states, edge cases, validation, UI-only flows |
| `frontend/e2e/real-backend/` | Real backend on localhost (via `make test-e2e`) | Happy path of any `/api/*` surface, auth headers, polling correctness, envelope shape |

**Hard rule** (enforced at spec, coder, and reviewer levels — PP15, TD03, validate-spec Category 10): every vertical-slice story MUST add or extend a spec in `e2e/real-backend/` covering at least the happy path. Any story touching fetch / polling / auth-header surfaces MUST also cover those surfaces in a real-backend spec.

## 2. Directory structure

```
frontend/
├── e2e/
│   ├── global.setup.ts              # Clerk signin + JWT swap, produces storageState
│   ├── global.teardown.ts           # Optional DB cleanup, Smocker reset
│   ├── fixtures/
│   │   ├── auth.ts                  # signInAsUserB, restoreUserA, SQL helpers
│   │   └── mock-responses.ts        # Centralised mock response builders (with Access-Control-Expose-Headers)
│   ├── smoke.spec.ts                # Navigation, auth gating, basic rendering (mocked)
│   ├── feature-<name>.spec.ts       # Per-feature mocked specs
│   └── real-backend/
│       ├── smoke-real.spec.ts       # Basic real-backend flow (sign in, load dashboard, assert data)
│       └── <feature>-real.spec.ts   # Per-surface real-backend specs for PP15 compliance
├── playwright.config.ts
└── playwright/.clerk/
    └── user-a.json                  # storageState artifact (gitignored, regenerated by global.setup.ts)
```

## 3. `playwright.config.ts` — canonical shape

```typescript
import { defineConfig, devices } from '@playwright/test';
import { config as dotenvConfig } from 'dotenv';

dotenvConfig({ path: '.env.local' });

const TARGET_URL = process.env.E2E_BASE_URL ?? 'http://localhost:3100';

export default defineConfig({
  testDir: './e2e',
  fullyParallel: false,                    // Prevent Clerk state collisions
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 2 : 0,
  workers: 1,                              // Single worker: deterministic, avoids rate limits
  reporter: process.env.CI ? [['github'], ['html']] : 'html',

  use: {
    baseURL: TARGET_URL,
    trace: 'on-first-retry',
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
  },

  // Global setup produces storageState consumed by all authenticated projects
  globalSetup: require.resolve('./e2e/global.setup.ts'),

  projects: [
    {
      name: 'chromium',
      use: {
        ...devices['Desktop Chrome'],
        storageState: 'playwright/.clerk/user-a.json',
      },
    },
  ],

  // CI: spin up production server; local: assume dev server already running
  webServer: process.env.CI
    ? {
        command: 'npm start',
        url: TARGET_URL,
        timeout: 60_000,
        reuseExistingServer: false,
      }
    : undefined,
});
```

**Key choices and why:**

- `fullyParallel: false` + `workers: 1` — Clerk session state is global per browser instance; parallelism causes cross-test auth pollution (PC01, PC02).
- `retries: 2` in CI — tolerates rare network blips without hiding real failures; use with `trace: 'on-first-retry'` to debug flakes.
- `npm start` (not `npm run dev`) in CI — avoids on-demand compilation timing out Clerk JS bootstrap (PP08).
- `storageState: 'playwright/.clerk/user-a.json'` — pre-authenticated session produced by `global.setup.ts`.

## 4. `global.setup.ts` — the JWT swap canonical implementation

The **single most load-bearing file** in the Playwright setup. Gets it wrong and every authenticated test fails with 401.

**What it does, in order:**

1. Launches a browser
2. Navigates to Clerk sign-in
3. Uses `@clerk/testing` to inject a Testing Token and sign in as `user-a`
4. **Swaps** the Testing Token for a real long-lived JWT via `createClerkClient().sessions.getToken(sessionId, 'playwright-ci')` — this is the fix for PP12
5. Replaces the `__session` and `__clerk_db_jwt` cookies in the storage state
6. Writes `playwright/.clerk/user-a.json`

**Prerequisites in the Clerk dev instance** (one-time setup, document in `clerk-patterns.md`):

- A JWT template named **`playwright-ci`** with lifetime **3600 seconds**
- Claims: include `sub` (user id) — backend RLS depends on this

**Canonical code:**

```typescript
// e2e/global.setup.ts
import { chromium, type FullConfig } from '@playwright/test';
import { clerkSetup, clerk } from '@clerk/testing/playwright';
import { createClerkClient } from '@clerk/backend';
import * as fs from 'node:fs';
import * as path from 'node:path';

const CLERK_SECRET_KEY = process.env.CLERK_SECRET_KEY!;
const CLERK_PUBLISHABLE_KEY = process.env.NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY!;
const USER_A_EMAIL = process.env.E2E_USER_A_EMAIL!;
const USER_A_PASSWORD = process.env.E2E_USER_A_PASSWORD!;
const STORAGE_STATE_PATH = path.resolve('playwright/.clerk/user-a.json');

export default async function globalSetup(_config: FullConfig) {
  if (!CLERK_SECRET_KEY || !CLERK_PUBLISHABLE_KEY) {
    throw new Error('CLERK_SECRET_KEY and NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY required');
  }

  await clerkSetup();
  fs.mkdirSync(path.dirname(STORAGE_STATE_PATH), { recursive: true });

  const browser = await chromium.launch();
  const context = await browser.newContext();
  const page = await context.newPage();

  // Sign in with Testing Token (page-level auth works, client-side getToken() does NOT)
  await page.goto('/sign-in');
  await clerk.signIn({
    page,
    signInParams: { strategy: 'password', identifier: USER_A_EMAIL, password: USER_A_PASSWORD },
  });
  await page.waitForURL((url) => !url.pathname.startsWith('/sign-in'), { timeout: 15_000 });

  // Extract session id
  const sessionId = await page.evaluate(() =>
    (window as any).Clerk?.session?.id as string | undefined
  );
  if (!sessionId) throw new Error('Clerk session id missing after signIn');

  // SWAP: Testing Token → long-lived JWT
  const backend = createClerkClient({ secretKey: CLERK_SECRET_KEY });
  const { jwt } = await backend.sessions.getToken(sessionId, 'playwright-ci');
  if (!jwt) throw new Error('Failed to obtain playwright-ci JWT');

  // Persist storage state, then patch cookies with the real JWT
  await context.storageState({ path: STORAGE_STATE_PATH });
  const state = JSON.parse(fs.readFileSync(STORAGE_STATE_PATH, 'utf-8'));
  for (const cookie of state.cookies) {
    if (cookie.name === '__session' || cookie.name === '__clerk_db_jwt') {
      cookie.value = jwt;
      cookie.expires = Math.floor(Date.now() / 1000) + 3300; // stay short of 3600 to avoid edge races
    }
  }
  fs.writeFileSync(STORAGE_STATE_PATH, JSON.stringify(state, null, 2));

  await browser.close();
}
```

**If you deviate from this pattern, you will spend hours debugging 401s.** The JWT swap is non-obvious; the `@clerk/testing` docs do not highlight it.

## 5. Spec structure — mocked variant (`e2e/<name>.spec.ts`)

```typescript
import { test, expect } from '@playwright/test';

test.describe('item creation — mocked', () => {
  test('happy path creates an item', async ({ page }) => {
    // Centralised mock response (includes Access-Control-Expose-Headers per PP14)
    await page.route('**/api/items', async (route) => {
      // Assert auth header attached (PP15 / PC06 compliance)
      expect(route.request().headers()['authorization']).toMatch(/^Bearer /);

      if (route.request().method() === 'POST') {
        await route.fulfill({
          status: 201,
          headers: {
            'Content-Type': 'application/json',
            'Access-Control-Expose-Headers': 'X-Correlation-Id',
            'X-Correlation-Id': 'test-corr-1',
          },
          body: JSON.stringify({ data: { id: 'item-1', title: 'Sample' } }),
        });
      }
    });

    await page.goto('/items/new');
    await page.getByLabel(/^Title$/).fill('Sample');     // Exact label (PP09)
    await page.getByRole('button', { name: 'Create', exact: true }).click();

    await expect(page.getByText('Sample', { exact: true })).toBeVisible({ timeout: 5_000 });
  });
});
```

**Required pattern elements:**

1. `route.request().headers()['authorization']` assertion (PC06)
2. `Access-Control-Expose-Headers` for any custom response header (PP14)
3. `{ exact: true }` on short text and role names (PP03, PP09, PP10)
4. `toBeVisible({ timeout })` with explicit timeout, not `waitForTimeout` (PP01)

## 6. Spec structure — real-backend variant (`e2e/real-backend/<name>-real.spec.ts`)

Used for the happy path of any `/api/*` surface (PP15). Differences from mocked:

- **No `page.route()`** on endpoints under test
- Asserts on **shape**, not specific upstream values (resilient to external flakiness)
- Uses SQL helpers to verify DB state (PP04 — never `page.request`)
- Expects backend to be running (`make test-e2e` orchestrates)

```typescript
import { test, expect } from '@playwright/test';
import { getItemCountForUser, USER_A_ID } from '../fixtures/auth';

test.describe('item creation — real backend', () => {
  test('creating an item persists to DB and renders in the list', async ({ page }) => {
    const initialCount = await getItemCountForUser(USER_A_ID);

    await page.goto('/items/new');
    await page.getByLabel(/^Title$/).fill('Real sample');
    await page.getByRole('button', { name: 'Create', exact: true }).click();

    // Wait for the backend to confirm persistence, not for an arbitrary timeout
    await page.waitForResponse(
      (r) => r.url().includes('/api/items') && r.request().method() === 'POST' && r.ok(),
      { timeout: 15_000 }
    );

    // Verify DB state via SQL helper (PP04 — NOT page.request)
    await expect
      .poll(async () => getItemCountForUser(USER_A_ID), { timeout: 10_000 })
      .toBeGreaterThan(initialCount);

    await expect(page.getByText('Real sample', { exact: true })).toBeVisible({ timeout: 10_000 });
  });
});
```

**What this catches that mocks don't:**

- Missing `Authorization` header (fetch arrives at the backend, backend rejects, 401 propagates back)
- Request shape drift (Zod/validation on the Go side)
- RLS misconfiguration (backend accepts but user B can see user A's row, caught via SQL helpers)
- Timing races in polling (`waitForResponse` exposes the real cadence)

## 7. Fixtures — `e2e/fixtures/auth.ts`

Centralises auth-related test utilities:

```typescript

import { clerk, clerkSetup } from '@clerk/testing/playwright';
import { createClerkClient } from '@clerk/backend';
import { Pool } from 'pg';

export const USER_B_ID = process.env.E2E_USER_B_CLERK_ID!;

// Superuser SQL helper — bypasses RLS, used for verifying DB state
export async function getItemCountForUser(userId: string): Promise<number> {
  const { rows } = await pgPool.query<{ c: string }>(
    'SELECT COUNT(*)::text as c FROM items WHERE user_id = $1',
    [userId]

  return Number(rows[0].c);
```

## 8. Test-auth mode (local only, not CI)

When running `make dev-offline`:

- `NEXT_PUBLIC_ENABLE_TEST_AUTH=true` on the frontend
- `ENABLE_TEST_AUTH=true` on the backend
- Frontend uses a `TestAuthProvider` instead of `ClerkProvider`
- All `fetch('/api/...')` attach `X-Test-User-Id: <uuid>` header instead of `Authorization: Bearer <jwt>`

**Playwright in test-auth mode is NOT the default.** It exists for:
- Debugging client-side rendering without Clerk JS loaded
- Iterating on UI without round-tripping through Clerk's signin flow

**Never write a CI spec that asserts `X-Test-User-Id`** (PC07). CI runs in real-Clerk mode; any such assertion is either wrong or silently rewritten by the coder.

If a project needs both branches exercised in CI, it's a dual-project Playwright config (one `real-clerk` + one `test-auth`). This is a feature, not a default — explicit spec-writer decision required.

## 9. Commands

From `Makefile`:

```makefile
# Local mocked E2E (fast iteration, no backend needed)
test-e2e-mocked:
	cd frontend && npx playwright test --grep-invert "real-backend"

# Full E2E: docker-compose stack + real backend + Smocker for externals
test-e2e:
	docker compose up -d postgres minio smocker
	# Apply migrations
	# Start backend with ENABLE_TEST_AUTH=false
	# Start frontend with npm start
	# Seed Smocker scenarios from e2e/fixtures/smocker/*.yml
	cd frontend && npx playwright test

# Single spec iteration
test-e2e-spec:
	cd frontend && npx playwright test $(SPEC) --debug
```

CI (`.github/workflows/ci.yml`) uses `make test-e2e` — no separate recipe.

## 10. Anti-flakiness checklist (reviewer-enforced)

A Playwright spec is merge-ready when ALL of these hold:

- [ ] No `waitForTimeout()` anywhere (PP01) — use `waitForResponse()` + `expect.poll()`
- [ ] No `test.describe.serial()` without explicit justification in a comment (PP02)
- [ ] `{ exact: true }` on every `getByText()` and `getByRole({ name })` with short text (PP03, PP09, PP10, PP11)
- [ ] Every authenticated endpoint mocked has a `route.request().headers()['authorization']` assertion (PC06)
- [ ] Every custom response header mocked has `Access-Control-Expose-Headers` (PP14)
- [ ] DB verification via `fixtures/auth.ts` SQL helpers, never `page.request` (PP04)
- [ ] `beforeEach(cleanup)` — no dependency on previous test state
- [ ] Isolated `browser.newContext()` for User B, followed by `context.close()` + `restoreUserA()` (PC01)
- [ ] No bare `Makefile:<line-n>` or environment-specific magic numbers in the spec — use env vars or constants in `fixtures/`

## 11. Troubleshooting

| Symptom | Likely cause | Where to look |
|---|---|---|
| All tests 401 after signin | JWT swap not happening | `global.setup.ts` step 4; check `playwright-ci` template exists in Clerk dev |
| Tests pass locally, fail in CI | `waitForTimeout` or `npm run dev` in CI | PP01, PP08 |
| Custom header `null` in JS | Missing `Access-Control-Expose-Headers` | PP14, backend CORS middleware |
| Auth works on page load, fails on fetch | Testing Token still active | PP12 — confirm JWT swap executed |
| Random test ordering cascades | `serial` mode or shared state between tests | PP02; add `beforeEach` cleanup |
| `getByText` strict-mode violation | Substring collision | PP03 — add `{ exact: true }` or use `getByRole` |
| User B test affects subsequent tests | `signOut()` without context isolation | PC01, PC05 — wrap in `signInAsUserB`/`restoreUserA` |
| Middleware not enforcing auth | `middleware.ts` at project root, not `src/` | PP13 |
| Test passes on mock but regresses in prod | Missing real-backend coverage | PP15 — add spec to `e2e/real-backend/` |
