# Parallel Worktree Protocol вҖ” N Agents on One Repo

> **Why this exists:** Claude Code's `Agent({isolation: "worktree"})` mode already creates per-agent git worktrees. But every worktree's dev stack tries to bind the same host ports (8080 backend, 3000 frontend, 5432 Postgres, 8082 Smocker), so only one agent can run an end-to-end test cycle at a time. This protocol defines the framework convention that makes N agents work in parallel without port conflicts, shared state corruption, or cleanup leaks.

The protocol is **opt-in per project**: projects that don't want parallel agents keep their existing dev workflow unchanged. The opt-in cost is a refactor of the project's `docker-compose.yml` to follow the convention below, plus adoption of the `kiat-worktree` CLI in the dev workflow.

---

## 1. Goals & non-goals

### Goals

| # | Goal |
|---|---|
| G1 | N agents run their dev stack in parallel on one developer machine with zero host port collisions. |
| G2 | Each agent's stack is **fully isolated**: independent network, independent volumes, independent migrations state. Crashing one agent's stack must not affect the others. |
| G3 | The agent discovers its own port allocation without manual configuration вҖ” single source of truth at `.worktree.env`. |
| G4 | Stack lifecycle is tied to worktree lifecycle: tearing down a worktree tears down its compose stack (no orphan containers, no orphan volumes). |
| G5 | Coexists with single-worktree mode вҖ” a developer working on the main checkout without worktrees keeps default ports `(8080, 3000, ...)` and never sees the protocol. |
| G6 | Framework-generic вҖ” the protocol does not assume Robia's stack. Every Kiat project (any language, any service set) can adopt it. |

### Non-goals

| # | Non-goal |
|---|---|
| NG1 | **CI/CD parallelism.** CI uses fresh checkouts in ephemeral runners, single environment per run вҖ” no port collision possible. This protocol is dev-only. |
| NG2 | **Centralized port registry / database.** No persistent state beyond per-worktree `.worktree.env` files. State lives where it's used. |
| NG3 | **Cross-worktree communication.** Agents in worktree A do not talk to services in worktree B. Each stack is its own universe. |
| NG4 | **Resource limits / quotas.** N stacks consume N Г— RAM. The framework documents the cost and provides `kiat-worktree list` to see usage; it does not enforce a ceiling. |
| NG5 | **Production parity.** This is a dev workflow. Production deployment configuration is unaffected. |
| NG6 | **Inter-agent rate limiting on shared external APIs.** If two worktrees both hit the real INPI sandbox, the agents must coordinate via Smocker or by sequencing the work. Out of scope here. |

## 2. Invariants

The protocol guarantees the following invariants вҖ” projects MUST preserve them when adopting.

| # | Invariant | Enforcement |
|---|---|---|
| I1 | A worktree's compose project name is unique and deterministic from its worktree-id. | `name: kiat-${WORKTREE_ID}` at the top of `docker-compose.yml`. |
| I2 | Backing services (databases, message brokers, fixture servers) **never publish a host port**. They communicate over the worktree's compose network only. | `expose:` clauses, never `ports:`, for any non-binary service. |
| I3 | App binaries (the things a human/agent connects to from outside the stack вҖ” backend HTTP, frontend HTTP, debug ports) **publish exactly one host port each**, computed from the worktree-id. | `ports: ["${KIAT_BACKEND_PORT}:8080"]`, env-driven. |
| I4 | Volumes are namespaced by worktree-id. | Named volumes: `postgres_data_${WORKTREE_ID}`. Bind mounts use absolute worktree paths. |
| I5 | The `.worktree.env` file is the **only** source of truth for port allocation. | Generated by `kiat-worktree create`. Read by compose (`env_file:`), Makefile, Playwright config, agent prompts. |
| I6 | Tearing down a worktree (git worktree remove) is preceded by tearing down its compose stack. | `kiat-worktree down` is mandatory before removal; `kiat-worktree gc` reaps orphans if forgotten. |
| I7 | A worktree-id never collides with an active worktree's id. | `kiat-worktree create` probes the port set and retries on collision (hash + linear probing). |

Violating any invariant means a project may *appear* to work in single-worktree mode and fail silently when a second agent is spawned. CI must include a lint that checks I1, I2, I3, I4, I6.

**Key observations:**

- Backing services (`pg`, `smocker`) have **no host port** вҖ” they are reachable only from within their own compose network via service-name DNS (`postgres:5432`, `smocker:8082`).
- App binaries (`bknd`, `frntd`) publish exactly one host port each, computed from the worktree-id.
- The two compose networks are isolated вҖ” `kiat-abc123_default` and `kiat-def456_default` cannot see each other. Two Postgres containers run concurrently with identical internal config; their data lives in two independent named volumes.

**Consequence:** native test runners that connect to the dev DB will not work under this protocol *if* the DB has no host port. Three resolutions, in order of preference:

1. **Run tests inside the stack** вҖ” `docker compose exec backend go test ./...`. Cleanest, matches CI behavior.
2. **Use Testcontainers** вҖ” tests spin up their own ephemeral DB per run. Standard pattern; no dev-stack dependency.
3. **Add the DB to the "exposed" tier** вҖ” opt out of I2 for this project. Loses host-port-uniqueness across worktrees for the DB specifically (collisions resurface). Document the tradeoff in the project's `delivery/specs/testing.md`.

The protocol recommends path 1 + path 2 for the testing flow.

## 4. Port allocation

### 4.1 Algorithm

```python
def allocate_slot(worktree_id: str, occupied_slots: set[int]) -> int:
    """Hash + linear probing. O(1) average for low load factor."""
    base = int(worktree_id[:6], 16) % 100   # 100 slots
    slot = base
    while slot in occupied_slots:
        slot = (slot + 1) % 100
        if slot == base:
            raise RuntimeError("all 100 slots in use вҖ” tear down a worktree first")
    return slot
```

`worktree_id` is the alphanumeric suffix of the worktree directory (e.g., `a325182a`). The Claude Code harness generates it; the protocol does not control its format.

### 4.2 Port layout

| Service class | Port formula | Range | Notes |
|---|---|---|---|
| Backend app | `18000 + slot` | 18000вҖ“18099 | One per worktree. Internal container port is project-defined (typically 8080); only the host mapping varies. |
| Frontend app | `13000 + slot` | 13000вҖ“13099 | Idem. |
| Backend debug (delve / pprof / etc.) | `19000 + slot` | 19000вҖ“19099 | Optional; activated only when project's `KIAT_ENABLE_DEBUG_PORT=1`. |
| Smocker admin UI (if exposed) | `18800 + slot` | 18800вҖ“18899 | Optional; default = not exposed. |
| Reserved | `18100вҖ“18799`, `13100вҖ“18999`, `19100+` | вҖ” | Future protocol extensions. Project-specific extra services must claim ranges outside `[13000, 19999]` and document them in their own spec. |

Main-checkout (no worktree, `.worktree.env` absent) keeps **default ports** (`8080`, `3000`, etc.). The compose file uses `${KIAT_BACKEND_PORT:-8080}` style defaults, so the same file works in both modes.

### 4.3 Collision resolution

Two worktrees with hash colliding into the same slot are detected at `kiat-worktree create` time:

1. The new worktree probes for `lsof -i :${slot_base + base}` (and the other slot ports) вҖ” if any port is in use, advance to the next slot.
2. After 100 probes without success, fail with a message pointing to `kiat-worktree list` and `kiat-worktree gc`.

No persistent registry. State is reconstructed each time from observed open ports + existing `.worktree.env` files.

### 4.4 Why not auto-assign (`ports: ["8080"]` without left side)?

Docker can auto-assign a random host port. This was rejected because:

- The port changes every time the stack is recreated вҶ’ agents that cached the port (in Playwright config, in a saved curl command, in `delivery/metrics/`) point to the wrong stack after restart.
- Discovery via `docker compose port` adds a runtime call to every dev command.
- The hash-derived port is stable for the worktree's lifetime, which is what tooling and humans expect.

Auto-assignment remains a valid alternative for projects that want truly stateless allocation вҖ” document the choice in `delivery/specs/parallel-worktree.md` and skip the `kiat-worktree create` step.

## 5. The `.worktree.env` contract

Single source of truth for the worktree's identity and port allocation.

### 5.1 File location

`<worktree-root>/.worktree.env` вҖ” a sibling of `docker-compose.yml`. Gitignored by Kiat-conformant projects (it's per-worktree, never committed).

### 5.2 File content

```bash
# Generated by kiat-worktree create on 2026-05-14T15:42:11Z.
# DO NOT EDIT вҖ” recreate the worktree if the slot must change.

KIAT_WORKTREE_ID=abc123
KIAT_SLOT=17
KIAT_BACKEND_PORT=18017
KIAT_FRONTEND_PORT=13017
KIAT_BACKEND_DEBUG_PORT=19017     # populated only if project opts in
KIAT_SMOCKER_ADMIN_PORT=18817     # populated only if project opts in

# Compose project name вҖ” locks the network + volume namespace.
# Used by docker compose via COMPOSE_PROJECT_NAME.
COMPOSE_PROJECT_NAME=kiat-abc123
```

### 5.3 How it is loaded

| Consumer | Mechanism |
|---|---|
| `docker compose` (variable substitution at parse time) | Loaded via `docker compose --env-file .worktree.env <cmd>` вҖ” the `kiat-worktree up`/`down` wrappers pass this flag automatically. Substitutes `${KIAT_BACKEND_PORT}`, `${KIAT_WORKTREE_ID}`, `${COMPOSE_PROJECT_NAME}` in the compose file. Do **not** list `.worktree.env` in any service's per-service `env_file:` вҖ” it has no value to the binary process and would fail-parse in main-checkout mode where the file doesn't exist. |
| Project Makefile | `include .worktree.env` at the top, or `export $(shell cat .worktree.env \| xargs)`. Wrap behind `ifneq ($(wildcard .worktree.env),)` to no-op in main mode. |
| Playwright config | `process.env.KIAT_FRONTEND_PORT` вҖ” config file reads it, falls back to `3000`. The env is exported by the dev shell (`eval $(kiat-worktree env)`) or by the wrapping Makefile target. |
| Backend / frontend dev binary | The binary does NOT read `.worktree.env` directly. It reads `PORT` / `DATABASE_URL` / project secrets from its container env, which comes from per-service `environment:` and per-service `env_file: [.env]` in the compose file вҖ” entirely separate from `.worktree.env`. |
| Agent prompts (Team Lead, coders) | Agents read `.worktree.env` at session start to set `BASE_URL` for any HTTP smoke test. Loaded by parsing the file directly (single-purpose env loader, no shell injection). |

### 5.4 What it is NOT

- Not a project config file. Project secrets and project env vars live in `.env` / `.env.example` as before. `.worktree.env` is strictly identity + ports.
- Not portable. Two worktrees with the same `.worktree.env` would conflict. The file MUST be regenerated by `kiat-worktree create` per worktree.

## 6. Compose file convention

A project's `docker-compose.yml` opts into the protocol by following four rules.

### 6.1 Rule 1 вҖ” Project name from env

```yaml
# docker-compose.yml вҖ” top-level
name: ${COMPOSE_PROJECT_NAME:-kiat-default}
```

Falls back to `kiat-default` in non-worktree mode (single-stack on the main checkout).

### 6.2 Rule 2 вҖ” Two service classes

| Class | Pattern | Rationale |
|---|---|---|
| **Backing service** (DB, cache, fixture server, message broker, etc.) | `expose: [...]`, no `ports:` | Internal-only; reachable from other containers via service-name DNS. Zero host-port footprint = zero collision possibility. |
| **App binary** (backend HTTP, frontend HTTP, agent-facing tooling) | `ports: ["${KIAT_X_PORT:-default}:internal"]` | Reachable from host on a per-worktree-unique port; falls back to default in non-worktree mode. |

### 6.3 Rule 3 вҖ” Volumes namespaced

```yaml
volumes:
  postgres_data:
    name: postgres_data_${KIAT_WORKTREE_ID:-default}
  smocker_history:
    name: smocker_history_${KIAT_WORKTREE_ID:-default}
```

Even though compose's default project-scoped volume naming would already isolate by project name, explicit `name:` makes the worktree affiliation visible in `docker volume ls` for ops debugging.

### 6.4 Rule 4 вҖ” Single network

```yaml
networks:
  default:
    name: kiat-${KIAT_WORKTREE_ID:-default}_default
```

Single isolated network per worktree. Service-to-service communication uses service-name DNS (`postgres:5432`). No `network_mode: host`, no shared external networks.

### 6.5 Reference compose

See `.claude/templates/docker-compose.parallel-worktree.example.yml` for a fully annotated reference. Projects should copy + adapt, not vendor verbatim вҖ” service lists differ.

## 7. Tooling: `kiat-worktree` CLI

A stdlib-only Python script at `.claude/tools/worktree.py`. Invoked via wrapper alias `kiat-worktree`.

### 7.1 Commands

| Command | Effect |
|---|---|
| `kiat-worktree create <branch-name>` | Creates a git worktree at `.claude/worktrees/agent-<id>/`, allocates a slot, writes `.worktree.env`, does NOT bring the stack up. Prints next steps. |
| `kiat-worktree up [--detach]` | `docker compose up --build` inside the current worktree. Reads `.worktree.env`. Waits for healthchecks (with timeout). |
| `kiat-worktree down [--volumes]` | `docker compose down`. With `--volumes`, removes named volumes too. |
| `kiat-worktree list [--json]` | Scans `.claude/worktrees/*/` for `.worktree.env` files, queries Docker for stack status, prints a table: id, slot, ports, branch, RAM, container count. |
| `kiat-worktree gc [--dry-run]` | Reaps orphan Docker objects (networks, volumes, containers) whose `kiat-<id>` project no longer has a corresponding worktree directory. |
| `kiat-worktree exec <cmd...>` | Shorthand for `docker compose exec <primary-service> <cmd>`. Primary service name is read from `.kiat-worktree-config.toml` if present, defaults to `backend`. |
| `kiat-worktree env` | Prints the current `.worktree.env` content to stdout (for sourcing: `eval $(kiat-worktree env)`). |
| `kiat-worktree remove` | Combines `kiat-worktree down --volumes` + `git worktree remove --force`. Single-command teardown. |
| `kiat-worktree lint [--strict]` | Validates the project's `docker-compose.yml` against the four rules from В§6 (env-driven name, no-host-port on backing services, env-driven ports on app services, namespaced volumes/networks). Exits 0 with warnings by default; `--strict` makes any violation a non-zero exit (suitable for CI). |

### 7.2 Lifecycle integration with agents

Claude Code's `Agent({isolation: "worktree"})` is opaque to the harness вҖ” it creates the worktree but does not run the `up` step. The protocol adds a **session-start hook** in the project's `.claude/settings.json`:

```json
{
  "hooks": {
    "SessionStart": [
      {
        "command": "test -f .worktree.env && kiat-worktree up --detach || true"
      }
    ]
  }
}
```

In a worktree session, the hook runs `kiat-worktree up` on entry. In the main checkout (no `.worktree.env`), the hook is a no-op and standard `make dev` works as before.

Teardown: when the parent session calls `Agent` with isolation worktree and the agent makes no changes, Claude Code auto-cleans up вҖ” but doesn't know about the compose stack. The session's `Stop` hook handles it:

```json
{
  "hooks": {
    "Stop": [
      {
        "command": "test -f .worktree.env && kiat-worktree down --volumes || true"
      }
    ]
  }
}
```

If a session crashes before `Stop` fires, the next `kiat-worktree gc` run reaps the orphan stack.

### 7.3 Why Python stdlib only

Same rationale as `doc-audit.py` and `report.py` вҖ” no `pip install` step in any consumer project. The script's external surface is `subprocess.run(["docker", "compose", ...])` and `subprocess.run(["git", "worktree", ...])`.

## 8. Agent integration

The protocol is mostly invisible to coder agents вҖ” by the time they start, `.worktree.env` is populated and the stack is up. Three integration points matter:

### 8.1 Coder agents вҖ” base URL discovery

Both `kiat-backend-coder` and `kiat-frontend-coder` agent prompts gain a "Worktree awareness" section:

> If `.worktree.env` exists at the repo root, read `KIAT_BACKEND_PORT` and `KIAT_FRONTEND_PORT`. Use them as your `BASE_URL` for any smoke test, curl, or Playwright run. Never hardcode `localhost:8080` or `localhost:3000`.

This is a ~20-token addition to each coder agent. Implementation: each agent's "Step 1" reads `.worktree.env` into its working memory.

### 8.2 Playwright config вҖ” port from env

Project's `playwright.config.ts` (or equivalent):

```typescript
export default defineConfig({
  use: {
    baseURL: `http://localhost:${process.env.KIAT_FRONTEND_PORT ?? 3000}`,
  },
  webServer: process.env.KIAT_WORKTREE_ID
    ? undefined  // Stack is up via compose; Playwright connects to it.
    : { command: 'npm run dev', port: 3000, reuseExistingServer: true },
});
```

In worktree mode, Playwright assumes the stack is already up (`webServer: undefined`). In main-checkout mode, the standard `webServer` config applies.

### 8.3 Team Lead вҖ” context

`kiat-team-lead.md` gets a one-line addition to its Stage 1.2 (clean working tree gate) protocol:

> If running inside a worktree (`KIAT_WORKTREE_ID` set), include the worktree-id in the metrics rollup so reports can correlate failures with parallel-execution conditions.

The metrics events schema (v1.2 вҖ” minor revision) adds an optional `worktree_id` field to `story_rollup` and `story_escalated` events. Backward-compatible вҖ” old reports ignore the field.

## 9. Migration guide for projects

A project adopts the protocol in **five steps**, each independently revertible.

### Step 1 вҖ” Audit current compose file

Identify, for each service:
- Does it expose a host port today? If yes, why? If "for tests", does that test need to be migrated to `docker compose exec` or Testcontainers?
- Does it use named volumes? Bind mounts? Are paths absolute or relative?

Tag each service as `backing` (DB, cache, fixture server) or `app` (backend HTTP, frontend HTTP).

### Step 2 вҖ” Refactor compose to env-driven

Apply the four rules from section 6:
- Top-level `name: ${COMPOSE_PROJECT_NAME:-kiat-default}`
- Backing services: remove `ports:`, add `expose:` if missing
- App services: change `ports: ["8080:8080"]` to `ports: ["${KIAT_BACKEND_PORT:-8080}:8080"]`
- Named volumes: add explicit `name:` with `${KIAT_WORKTREE_ID:-default}` suffix
- Single network with `name: kiat-${KIAT_WORKTREE_ID:-default}_default`

Validate: `docker compose config` in main checkout (no `.worktree.env`) вҖ” must produce the same effective config as before the refactor.

### Step 3 вҖ” Migrate native tests to in-stack or Testcontainers

For each test suite that currently connects to a service over `localhost:<host-port>`:
- **Preferred**: refactor to run inside the stack вҖ” `docker compose exec backend go test ./...`. Update Makefile target.
- **Alternative**: refactor to Testcontainers вҖ” tests own their DB lifecycle, dev stack is unaffected.

This is the step that usually unblocks adoption. Test-on-host expectations are the most common reason backing services need host ports.

### Step 4 вҖ” Install `kiat-worktree` tooling

Symlink or vendor `kiat/.claude/tools/worktree.py` as `kiat-worktree` in the project's PATH (developer convention вҖ” often via `direnv` or a shell rc snippet). Add a `make worktree-up` / `make worktree-down` wrapper to the project's Makefile if Makefile is the canonical entry point.

### Step 5 вҖ” Project-side spec

Write `delivery/specs/parallel-worktree.md` in the project (project-scoped):
- The list of services and their class (backing vs app)
- The chosen port ranges if the project deviates from the framework defaults
- The chosen test execution path (in-stack vs Testcontainers)
- Project-specific opt-outs (e.g., "Postgres exposes a host port because legacy tests depend on it вҖ” collision happens beyond 1 worktree, fix tracked in epic-XX")

Link this from the project's `CLAUDE.md` under "Where to find project rules".

### Step 6 вҖ” Verify

```bash
# In a fresh terminal
$ kiat-worktree create test-parallel-1
вң“ Created worktree at .claude/worktrees/agent-a3f291/
вң“ Allocated slot 41 (backend 18041, frontend 13041)
вң“ Wrote .worktree.env
вҶ’ Next: cd .claude/worktrees/agent-a3f291 && kiat-worktree up

$ cd .claude/worktrees/agent-a3f291 && kiat-worktree up
[builds, starts, waits for healthchecks]
вң“ Stack kiat-a3f291 up (backend http://localhost:18041, frontend http://localhost:13041)

# In a second terminal
$ kiat-worktree create test-parallel-2
вң“ Created worktree at .claude/worktrees/agent-b7e482/
вң“ Allocated slot 78 (backend 18078, frontend 13078)
...

$ kiat-worktree list
ID       SLOT  BACKEND  FRONTEND  BRANCH              STATUS  RAM
a3f291   41    18041    13041     test-parallel-1     up      612MB
b7e482   78    18078    13078     test-parallel-2     up      587MB
```

Both stacks running. `curl localhost:18041/api/health` and `curl localhost:18078/api/health` hit different backends. Done.

## 10. Failure modes & edge cases

| # | Failure mode | Detection | Mitigation |
|---|---|---|---|
| F1 | Session crashes before `Stop` hook вҶ’ orphan compose stack | `kiat-worktree list` shows a stack whose worktree dir was removed by `git worktree remove --force` | `kiat-worktree gc` reaps orphans; suggest running it weekly |
| F2 | Hash collision after gc didn't run | `kiat-worktree create` returns "all 100 slots in use" | Run `kiat-worktree gc`; if still full, manually `kiat-worktree remove <id>` |
| F3 | Project compose file forgets a rule (e.g., uses `ports:` on Postgres) | Two worktrees fight for the same Postgres port; second `kiat-worktree up` fails | CI lint: `python3 .claude/tools/worktree.py lint` scans `docker-compose.yml` against the four rules; fails CI if any service violates I2/I3 |
| F4 | Backing service has no host port вҶ’ developer can't connect with their local DB client | "Permission denied" or "connection refused" on `psql -h localhost` | `kiat-worktree exec psql` shorthand; or temporary `docker compose up --service-ports postgres` for ad-hoc inspection |
| F5 | Two worktrees share a named volume by mistake | Data corruption вҖ” one stack reads what the other wrote | Explicit `name:` with `${KIAT_WORKTREE_ID}` suffix in each volume; CI lint also validates this |
| F6 | Docker Desktop runs out of memory | Random container kills, OOM in logs | `kiat-worktree list` shows per-stack RAM; document the recommended Docker Desktop memory limit (4Г— single-stack baseline for 4 concurrent worktrees) |
| F7 | Migrations race condition: two worktrees on the same branch run migrations against their own DBs simultaneously | No real race вҖ” each DB is independent вҖ” but the migration files in code are shared | Acceptable: each worktree's migration state is independent |
| F8 | External API rate limit (e.g., real INPI sandbox) | API returns 429 to whichever worktree calls last | Force agents to point at Smocker for external calls in worktree mode; document in the project spec that real-API mode is single-worktree-only |
| F9 | Stale `.worktree.env` after manual git worktree remove | `kiat-worktree list` reports the entry but `git worktree list` doesn't | `kiat-worktree gc` reconciles: removes the `.worktree.env` ref and tears down the matching compose stack |
| F10 | Developer copies a worktree directory instead of using `git worktree add` | Two dirs point at the same `COMPOSE_PROJECT_NAME` вҖ” second `up` reuses the first's stack | `kiat-worktree create` is the only entry point; CI lint warns if `.worktree.env` is checked into git |

## 11. Tradeoffs documented

| Choice | Alternative | Why we chose this |
|---|---|---|
| Hash-based deterministic slot | Sequential allocation via registry | No persistent state. The cost (small chance of collision вҶ’ linear probe) is cheaper than the cost of a registry file (race conditions, stale entries, gitignore discipline). |
| Internal-only backing services | Per-worktree host ports for all services | Halves the host-port surface. Eliminates the most common adoption blocker (port collision on DB). Forces tests into stack or Testcontainers вҖ” which is closer to CI behavior anyway. |
| Python stdlib tool | Bash, Go, Node | Consistent with existing Kiat tools (`doc-audit.py`, `report.py`). Zero install step in consumer projects. |
| Session hooks for lifecycle | Agent prompt step | Hooks are project-level and run in the harness вҖ” guaranteed to fire even if the agent crashes mid-prompt. Prompt-level lifecycle is fragile. |
| `.worktree.env` as single source of truth | Discovery via `docker compose port` at every use | Stability for the worktree's lifetime + zero runtime Docker calls in dev commands. Auto-discovery is documented as an opt-in alternative. |
| Range 18000-18099 + 13000-13099 | Port range from `/etc/services` (above 49152) | Avoids ephemeral port range on Linux (32768-60999 by default). Keeps numeric proximity to default 8080/3000 so the mental model is clear. |

## 12. Out of scope (deferred)

| # | Item | Reason |
|---|---|---|
| O1 | Resource limit enforcement (CPU/memory per stack) | YAGNI вҖ” `kiat-worktree list` shows usage; developers self-regulate. Revisit if observed pain. |
| O2 | Cross-host worktrees (remote dev machine) | Single-dev-machine assumption is core to the design. Remote dev is a different problem (devcontainers / DevPod / Coder). |
| O3 | macOS Docker Desktop performance tuning | Document only вҖ” bind-mount perf on Mac is a known sore spot, but tuning is OS-specific and outside the protocol. |
| O4 | Auto-pin Docker image digests across worktrees | The worktree just inherits the project's image pinning. Cross-worktree pin consistency is a project concern, not a protocol concern. |
| O5 | Visual dashboard for `kiat-worktree list` | Plain text is enough for now. A web UI is YAGNI until pain is observed. |
| O6 | Auto-recreate stack on `docker-compose.yml` change | Manual `kiat-worktree down && kiat-worktree up` is the contract. Watch loops add complexity for marginal value. |
| O7 | Agent-aware load balancer (route requests to least-loaded worktree) | Not a worktree concern вҖ” that's a multi-stack request routing topic, separate scope. |

## 13. Worked example: adopting on Robia

Robia's stack today (pre-protocol):

```yaml
# docker-compose.yml (current)
services:
  postgres:
    ports: ["5432:5432"]
  smocker:
    ports: ["8082:8082"]
```

Backend + frontend run natively (`go run`, `npm run dev`) on 8080 / 3000, reaching Postgres on `localhost:5432`. Migration to the protocol:

1. **Refactor compose:**
   ```yaml
   name: ${COMPOSE_PROJECT_NAME:-kiat-default}
   services:
     postgres:
       expose: ["5432"]               # was ports
     smocker:
       expose: ["8082"]               # was ports
     backend:                          # new вҖ” backend is now in-stack
       build: ./backend
       ports: ["${KIAT_BACKEND_PORT:-8080}:8080"]
       env_file: [".env"]              # .worktree.env is NOT here вҖ” see В§5.3
     frontend:                         # new вҖ” frontend is now in-stack
       build: ./frontend
       ports: ["${KIAT_FRONTEND_PORT:-3000}:3000"]
       env_file: [".env"]              # .worktree.env is NOT here вҖ” see В§5.3
   volumes:
     postgres_data:
       name: postgres_data_${KIAT_WORKTREE_ID:-default}
   networks:
     default:
       name: kiat-${KIAT_WORKTREE_ID:-default}_default
   ```

2. **Migrate tests:**
   - Venom suite already runs in-stack via `make test-venom` вҖ” no change.
   - Go unit tests use Testcontainers (`backend/internal/.../*_test.go`) вҖ” no change.
   - `make dev` currently launches binaries natively. After migration: `make dev` either calls `kiat-worktree up` (in worktree mode) or `docker compose up --build` (main mode). Either path now goes through compose.

3. **Test parallel:**
   ```bash
   # Terminal 1
   git worktree add .claude/worktrees/agent-story-01 chore/ultra-review-followup
   cd .claude/worktrees/agent-story-01
   kiat-worktree create story-01    # writes .worktree.env, slot 41
   kiat-worktree up
   # backend at :18041, frontend at :13041

   # Terminal 2
   git worktree add .claude/worktrees/agent-story-03 main
   cd .claude/worktrees/agent-story-03
   kiat-worktree create story-03    # writes .worktree.env, slot 78
   kiat-worktree up
   # backend at :18078, frontend at :13078
   ```

Two agents can now run end-to-end tests in parallel. Adoption effort estimate for Robia: **1 S story** (~1 j-h backend + ~0.5 j-h frontend test migration + ~0.5 j-h Makefile / hooks + ~0.5 j-h docs). Not on the cutover-followup critical path вҖ” schedule J+7 or later.
